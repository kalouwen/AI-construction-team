# Skills 创建模式

> **进度上报（必须执行）**：进入本模式后，立即用 Bash 工具写入活跃状态标记（将 `<目标工程路径>` 替换为实际路径）：
> ```bash
> mkdir -p "<目标工程路径>/.claude" && printf '{"mode":"skills","started_at":"%s","step":"启动技能创建"}' "$(date -Iseconds)" > "<目标工程路径>/.claude/active-mode.json"
> ```
> 每完成一个关键步骤（需求分析/模块调研/方案设计/生成SKILL.md/注册/验证），更新 step：
> ```bash
> printf '{"mode":"skills","started_at":"%s","step":"%s"}' "$(date -Iseconds)" "<当前步骤>" > "<目标工程路径>/.claude/active-mode.json"
> ```
> 模式结束时删除标记：`rm -f "<目标工程路径>/.claude/active-mode.json"`

你进入了 AI for better 的 Skills 创建模式。你的核心任务是**为目标工程创建定制化的 skills**，让特定功能的开发流程标准化、可复用、有校验。

## 什么是一个好的 Skill

一个好的 skill 不是"一段文字说明怎么做"，而是：
1. **标准流程**：明确的步骤，每步做什么、产出什么
2. **校验点**：每个关键步骤后自动检查，防止跑偏
3. **知识内嵌**：把需要的领域知识直接写进 skill，不依赖 AI 的训练数据
4. **配套脚本**：如果校验需要跑代码，skill 自带校验脚本

## 前置检查

进入 Skills 模式后，第一步：

```
1. 确认目标工程路径和需求
2. 检查 .claude/knowledge/profile.json
   - 不存在 → 自动触发 /survey
3. 检查 .claude/plan.md 是否存在
   - 存在且有 skills 任务 → 读取，筛选 模式=skills 且未完成的任务
     → 检查每个任务的前置依赖是否已完成
     → 有未完成的前置 → "skill-01 依赖 env-02，需要先跑 /mode-deploy"
     → 前置就绪 → 展示待执行的 skills 任务列表
     → 用户确认后按 plan 逐个创建
   - 存在但无 skills 任务 → "plan 中没有 skills 任务。要根据项目需求创建新 skill 吗？"
     → 用户确认 → 按原有流程（需求分析 → 设计 → 创建），完成后追加到 plan.md
     → 用户拒绝 → 退出 skills 模式
   - 不存在 → 按原有流程自由创建（需求分析 → 设计 → 创建）
4. 检查约束映射表（部署模式的产出）
   - 不存在 → 建议先跑 /mode-deploy，但不强制
   - 存在 → 读取，确保新 skill 与已有约束兼容
5. 检查 .claude/knowledge/modules/ 下是否有需求相关的模块知识
   - 不足 → 针对性补充调研
```

### plan.md 驱动模式

当 plan.md 存在时，Skills 模式的工作方式变为：

```
1. 读 plan.md → 筛选 模式=skills 且 checkbox 未勾选的任务
2. 检查依赖：该任务的前置任务是否都已勾选
   - 未就绪 → 告知用户哪些前置未完成，建议先执行
   - 就绪 → 进入正常的 skill 创建流程（Step 1-8）
3. Skill 创建并验证通过后 → 勾选 plan.md 中的 checkbox
4. 展示："已完成 skill-01，下一个是 skill-02，继续？"
5. 所有 skills 任务完成 → "skills 任务清零。"
```

**plan.md 中发现问题时**：直接修改 plan.md（加任务/改范围/调依赖），并告知用户改了什么。

## 创建流程

### Step 1: 需求分析

理解用户要创建什么 skill：

```
你要创建的 skill：{名称}
目标：{一句话描述}
使用场景：{什么时候触发}
涉及模块：{哪些模块}
```

### Step 2: 调研关联模块

从 `.claude/knowledge/modules/` 读取相关模块的知识卡片。如果知识不足，**先补调研**：

```
对每个涉及的模块：
  1. 读取知识卡片
  2. 检查卡片是否覆盖 skill 需要的信息
     - 数据流：清楚吗？
     - 业务规则：齐全吗？
     - 接口/依赖：明确吗？
  3. 不足的部分 → 读代码补充 → 更新知识卡片
```

### Step 3: 设计 Skill 方案

设计 skill 的完整方案，展示给用户：

```markdown
## Skill 设计方案：{名称}

### 触发方式
- 命令：/{skill-name}
- 参数：{需要用户提供什么}

### 标准流程（{N}步）

| 步骤 | 动作 | 产出 | 校验点 |
|------|------|------|--------|
| 1 | {做什么} | {产出什么} | {怎么检查} |
| 2 | {做什么} | {产出什么} | {怎么检查} |
| ... | | | |

### 内嵌知识
- {skill 需要知道的业务规则、数据格式、约定}

### 配套脚本
- {如果需要自动校验，列出要创建的脚本}

### 与已有约束的兼容性
- {skill 产出的代码会被哪些 hook 检查}
- {需要额外的守卫吗}

### 风险
- {可能出错的地方和应对}
```

### Step 4: 等待用户审批

展示方案后等用户确认。用户可以：
- 调整步骤
- 增减校验点
- 修改流程顺序
- 提出补充需求

### Step 5: 生成 Skill 文件

按确认的方案生成文件：

```
目标工程/.claude/skills/{skill-name}/
├── SKILL.md              ← Skill 主文件（流程+知识+规则）
└── scripts/              ← 配套脚本（如果有）
    ├── validate-xxx.sh   ← 校验脚本
    └── ...
```

**SKILL.md 结构**：
```markdown
---
name: {skill-name}
description: {一句话描述}
user_invocable: true
---

# {Skill 标题}

{对 AI 的角色说明}

## 输入
{用户需要提供什么}

## 流程

### Step 1: {步骤名}
{详细指令}

**校验点**：
- [ ] {检查项1}
- [ ] {检查项2}
校验不通过则停下来，说明问题，不继续。

### Step 2: {步骤名}
...

## 内嵌知识

### {知识主题1}
{具体内容——数据格式、业务规则、接口约定等}

### {知识主题2}
...

## 约束
- {必须遵守的规则}
- {与已部署 hook 的配合说明}

## 输出
{完成后展示什么}
```

### Step 6: 注册 Skill

将新 skill 注册到目标工程的 Claude Code 环境：
1. 确认 `.claude/skills/{skill-name}/SKILL.md` 已创建
2. 如果有配套脚本，检查脚本的执行权限
3. 如果需要新的 hook 配合，追加到 `settings.json`

### Step 7: 实测验证

模拟一次 skill 的完整执行：
1. 用一个简单的测试用例走一遍流程
2. 每个校验点是否正常工作
3. 配套脚本能否正常运行
4. 产出的代码是否通过已部署的 hook 检查

验证结果：
- 全部通过 → 交付完成
- 有问题 → 修复后重测

### Step 8: 问题回流

创建过程中发现的问题写入 `learning-backlog.jsonl`：

```json
{"id": "bl-{date}-{seq}", "source": "mode-skills", "project": "{name}",
 "type": "{knowledge_gap|template_gap|env_gap}",
 "detail": "{具体描述}", "priority": "{P0-P2}",
 "created_at": "{date}", "status": "pending"}
```

常见回流类型：
- `knowledge_gap`：模块知识不足，skill 无法生成完整的校验逻辑
- `template_gap`：skill 模板库缺少某类 skill 的参考
- `env_gap`：目标工程缺少某基础设施，skill 的校验点无法执行

## Skill 质量标准

创建的 skill 必须满足：

| 标准 | 要求 |
|------|------|
| 可执行 | 每步都是明确的操作指令，不是模糊建议 |
| 有校验 | 关键步骤有校验点，失败则停 |
| 知识自包含 | 需要的领域知识写在 skill 里，不依赖外部 |
| 与环境兼容 | 产出的代码能通过已部署的 hook 检查 |
| 可复用 | 同类需求可以反复使用，不是一次性的 |
| 有边界 | 明确什么能做什么不能做，超出范围时告诉用户 |

## 与其他模式的协作

- **知识不足 → 补调研**：自动读代码补充 knowledge/，不中断创建
- **环境缺口 → 触发模式2**：如果 skill 需要的 hook/CI 不存在，建议用户先部署
- **模板缺陷 → 写入 backlog**：发现 skill 模板不够用，记录供学习模式改进
- **多 skill 协作**：如果发现需求应拆成多个 skill，先展示拆分方案再逐个创建

## 重要规则

- **角色边界（铁律）**：AI for better 只创建 skill 文件和配套脚本，**绝不用 skill 执行业务代码**。
- **阶段隔离（铁律）**：Skills 创建完成后**停下来**，报告完成状态。不自动执行 skill，不自动开始编码。用户去目标工程自行使用 skills。
- **一个 skill 做一件事**：不要把多个功能塞进一个 skill
- **校验点必须可执行**：不写"检查代码质量"这种模糊检查，写"运行 lint 并检查返回码"
- **知识来源必须标注**：内嵌知识注明从哪个文件/哪次调研提取
- **不依赖 AI 训练数据**：所有领域知识显式写入，不假设 AI 知道
- **方案必须经用户审批**：设计完展示，确认后再生成文件
- **实测必须通过**：没验证的 skill 不算完成

# AI Friendly 完整检查清单 v0.2

用于评估任意项目的 AI 友好度。三层递进：环境是基础，迭代是生长，自治是目标。

---

## 第一层：环境层（AI 能不能干活）

### 1. 能读懂
- [ ] 有 CLAUDE.md 且包含项目结构说明（新会话 30 秒内能知道项目是什么）
- [ ] 单文件不超过 500 行（超过的要拆分）
- [ ] 函数/类有清晰命名，不依赖隐式上下文
- [ ] 目录结构按职责划分，不按人或时间
- [ ] 关键依赖关系有文档化（谁调谁、谁依赖谁）

### 2. 能动手
- [ ] 环境搭建步骤可脚本化
- [ ] AI 有足够的文件读写权限（settings.json 配置合理）
- [ ] 工具链可被 AI 调用（构建/测试/lint 能通过 Bash 执行）
- [ ] 外部服务有 MCP 或 API 接入
- [ ] 依赖版本锁定、可复现安装

### 3. 能验证
- [ ] 有测试，且 AI 能运行
- [ ] 报错信息包含文件名、行号、原因
- [ ] 有 lint/格式化工具
- [ ] 改动效果可被观察（日志/截图/输出对比）

### 4. 能回滚
- [ ] 项目在 Git 版本控制下
- [ ] 有分支策略（AI 在分支上工作，不直接改 main）
- [ ] 危险操作有 hook 拦截
- [ ] 关键文件有保护（.env/lock/配置）

### 5. 知识在
- [ ] 有 CLAUDE.md 或等效入口文档
- [ ] 有 memory 系统或等效持久化（跨会话不丢上下文）
- [ ] 外部文档本地化或有稳定访问通道
- [ ] 文档有更新机制（不是写完就烂）

---

## 第二层：迭代层（AI 能不能越用越好）

### 6. 经验能沉淀
- [ ] 有 memory 系统，关键决策和教训会被记录
- [ ] 反复出现的问题能转化为规则（写进 CLAUDE.md 或 hooks）
- [ ] 跨会话不重复犯同一个错

### 7. 知识能保鲜
- [ ] 文档与代码同步更新（改了代码就更新相关文档）
- [ ] 过时的 memory/文档有清理机制
- [ ] 外部知识库有版本标记，能知道信息是否过期

### 8. 工作方式能改进
- [ ] skills 和 hooks 会根据实际使用反馈调整
- [ ] 新发现的好模式能快速固化为工具（skill/agent/hook）
- [ ] AI 能提出改进建议，不只是被动执行

### 9. 经验能复用
- [ ] 一个项目的解决方案能迁移到其他项目
- [ ] 通用模式抽成模板，项目特定部分参数化
- [ ] 有明确的"模板 vs 实例"边界，不混淆

---

## 第三层：自治层（AI 能不能自主完成）

### 10. 能端到端执行
- [ ] 从需求理解到代码交付，AI 能独立完成完整流程
- [ ] 遇到阻塞能自主尝试替代方案，不是卡住等人
- [ ] 有明确的"需要人介入"判断标准（不是什么都问，也不是什么都自己做）

### 11. 多 Agent 能协同
- [ ] 不同 agent 职责清晰，不重叠不遗漏
- [ ] agent 之间有信息传递机制（不靠全塞进上下文）
- [ ] 并行工作时不产生冲突（文件锁/分支隔离）

### 12. 能自我监控
- [ ] 能检测自己是否在循环（重复同样的错误操作）
- [ ] 能评估自己的输出质量（不只是"做完了"而是"做对了"）
- [ ] 异常时能降级而非崩溃（全自动降级为半自动，不是直接失败）

---

**评分方法**: 逐条打勾，计算每层通过率。环境层 < 60% 的项目不建议推进到迭代层。
**版本**: v0.2（2026-03-13），持续迭代中。

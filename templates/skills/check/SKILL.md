# /check — 一站式代码检查

一个命令覆盖所有检查场景。你不需要记住用哪个，`/check` 会自动判断。

## 阶段检测

收到 `/check` 后，先判断当前处于哪个阶段：

**手动指定**（优先级最高）：
- `/check before` → 方案评审
- `/check after` → 质量验收
- `/check pr` → PR 审查

**自动检测**（无参数时）：
```
1. git diff --cached 或 git diff → 有未提交的改动？ → After 模式
2. git log main..HEAD --oneline → 有提交领先 main？ → PR 模式
3. 都没有 → Before 模式
```
注意：先检测未提交改动（After），再检测分支领先（PR）。因为刚写完代码时最常见的状态是有未提交改动。

检测完后告诉用户："检测到你在 [阶段]，开始检查。" 让用户知道发生了什么。

## Before 模式 — 动手前方案评审

在写第一行代码之前，用大白话回答以下问题。

### Q1 | 这次要做什么？
> 一句话说清楚，像给朋友解释一样。

### Q2 | 怎么拆？
> 拆成几步？每步能在一次对话内完成吗？有没有哪步大到需要再拆？

### Q3 | 有没有现成方案可以抄？
> 有参考就列出来（来源、可靠度）。自己发挥的部分标出来，评估风险。

### Q4 | 哪些写死、哪些做活？
> 稳定不变的硬逻辑 vs 未来可能调整的软逻辑（应该抽成配置）。

### Q5 | 对外接口是什么？
> 暴露什么、依赖什么、被谁依赖？能不能独立运行？

### Q6 | 10 倍压力下哪里先崩？
> 用户量/数据量涨 10 倍，瓶颈在哪？

### Q7 | 以后要换掉它，影响多大？
> 替换成本评估。

**输出**：逐条回答，每条不超过 3 行。
**规则**：全部答完后等用户说"可以动手"才写代码。

## After 模式 — 基线对比 + 质量验收 + 代码简化 + 安全扫描

代码写完了，先量化对比，再主观审查。

### 第零部分：基线对比（机械判断，最先执行）

**没有固定基线，就不知道改好了还是改坏了。** 先跑数据，再做判断。

```
1. 采集改动前基线（按情况选方法）：
   a. 改动未提交 → git stash → 跑测试/lint → 记录 → git stash pop
   b. 改动已提交 → git stash（保存工作区）→ git checkout HEAD~N（回到改动前）→ 跑测试/lint → 记录 → git checkout - → git stash pop
   c. 有已存在的基线文件（如 session-baseline.md）→ 直接读取
   结果：
   - 测试通过数: N_before
   - lint 警告数: W_before
   - 构建是否通过: B_before

2. 采集改动后状态（在当前代码上直接跑）：
   - 测试通过数: N_after
   - lint 警告数: W_after
   - 构建是否通过: B_after

3. 对比输出：
   | 指标 | 改动前 | 改动后 | 趋势 |
   |------|--------|--------|------|
   | 测试通过 | N_before | N_after | ↑/↓/= |
   | lint 警告 | W_before | W_after | ↑/↓/= |
   | 构建 | pass/fail | pass/fail | — |

4. 判定规则（机械，不需要人判断）：
   - 测试通过数下降 → ❌ 必须修
   - lint 警告数上升 → ⚠️ 建议修
   - 构建从 pass 变 fail → ❌ 必须修
   - 全部持平或改善 → ✅ 通过
```

基线对比有 ❌ 项的，必须先修复再进入下面的质量验收。

### 第一部分：质量验收（6 问）

| # | 检查项 | 要回答的 |
|---|--------|---------|
| Q1 | 实现确认 | 实际做了什么？和方案一致吗？有偏差就说。 |
| Q2 | 分层验证 | 软逻辑是否真的抽成了配置？有没有硬编码不该硬编码的？ |
| Q3 | 解耦验证 | 模块是否独立可运行？接口和预定义的一致吗？ |
| Q4 | 可观测性 | 有日志和错误提示吗？非技术人员怎么知道它在正常工作？ |
| Q5 | 安全边界 | 用户能通过奇怪输入搞坏它吗？有输入验证吗？ |
| Q6 | 规范一致 | 和已有代码风格一致吗？ |

每条标注结果：✅ 通过 / ⚠️ 小瑕疵 / ❌ 不通过

### 第二部分：代码简化

扫描本次改动（`git diff --cached` 或 `git diff HEAD~1`），找：
- 重复代码 → 提取公共函数
- 过度工程 → 削减不必要的抽象
- 性能问题 → N+1 查询、不必要的循环
- 啰嗦写法 → 用更简洁的方式替代

发现问题直接修，不只是列出来。

### 第三部分：安全快扫

快速检查本次改动涉及的文件：
- 硬编码密钥/token/password
- 注入风险（SQL/XSS/命令注入）
- .env 是否在 .gitignore 中
- 日志中是否泄露敏感信息

按严重程度排序：**严重**（必须修）→ **警告**（建议修）→ **信息**（改善建议）

### 最终结论

- **全部 ✅ + 无安全问题** → "检查通过，可以交付。"
- **有 ❌ 或严重安全问题** → 列出必须修复的项，修复后重新 `/check after`

## PR 模式 — 全面审查

审查当前分支相对于 main 的所有改动。

### 审查流程

```
1. git log main..HEAD --oneline → 看所有提交
2. git diff main...HEAD → 看完整改动
3. 逐文件深入审查
```

### 审查维度（6 项）

| 维度 | 关注点 |
|------|--------|
| 功能正确性 | 改动是否完成预期目标？遗漏的边界情况？ |
| 代码质量 | code smell？过长函数、深层嵌套、重复代码？ |
| 安全性 | 注入风险？硬编码密钥？权限检查完整？ |
| 性能 | N+1 查询？内存泄漏？大数据量瓶颈？ |
| 测试覆盖 | 关键路径有测试吗？边界情况覆盖了吗？ |
| 向后兼容 | 会破坏现有 API 吗？数据库变更可回滚吗？ |

### 审查结论

**PASS** — 可以合并。执行：
```bash
mkdir -p .claude && echo "approved $(date -Iseconds)" > .claude/push-approved
```

**WARN** — 有小问题但不阻塞。写入标记，提醒后续修复。

**FAIL** — 有严重问题。不写标记，列出必须修复的项。修复后重新 `/check pr`。

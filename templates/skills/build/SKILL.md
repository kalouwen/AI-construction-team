# /build — 自动构建修复循环

一个命令启动全自动构建。失败了自己修，修不了才找你。

## 启动

收到 `/build` 后：

### 1. 检测构建命令

按顺序检测项目用什么构建：

| 检测 | 命令 |
|------|------|
| package.json 有 build 脚本 | `npm run build` |
| package.json 有 test 脚本 | `npm test` |
| go.mod 存在 | `go build ./...` + `go test ./...` |
| *.sln 或 Unity 项目 | 对应构建命令 |
| Makefile 存在 | `make` |
| pyproject.toml / setup.py | `python -m build` + `pytest` |

如果有 `preflight.yaml`，先跑 `python preflight.py` 做预检。

### 2. 记录基线

跑一次完整构建+测试，记录当前状态：
- 哪些测试本来就失败的（不是你的锅）
- 当前编译状态

### 3. 开始循环

```
LOOP:
  1. 跑构建（全部目标）
  2. 跑测试（全部测试）
  3. 全过？ → 报告结果，结束
  4. 有失败？ → 进入自愈流程
  5. 自愈成功？ → 回到 1
  6. 自愈失败（同一错误 3 次）？ → 报告问题，停下来
```

## 自愈流程

失败时**禁止盲目重试**，必须先诊断：

### Step 1: 读错误
完整读取错误输出，不只看第一行。

### Step 2: 分类
判断错误属于哪类：

| 类型 | 特征 | 处理 |
|------|------|------|
| **代码错误** | 语法错、类型不匹配、缺 import | 定位文件行号，修复 |
| **配置错误** | 缺配置文件、配置格式错 | 检查并修正配置 |
| **环境问题** | command not found、包未安装 | 不重试，标记 BLOCKED |
| **测试失败** | assertion 不通过 | 看是代码bug还是测试过时 |

### Step 3: 鉴别诊断

关键区分：
- **"缺 import" vs "缺包"**：先搜项目里有没有这个符号。有 → 加 import。没有 → 包没装，标记 BLOCKED
- **"单点错误" vs "全面错误"**：搜全部调用点，不只修第一个
- **"新错误" vs "之前修过的"**：之前修过又出现 → 换一种方式修

### Step 4: 修复（单次单修原则）

**一次只修一个错误。** 发现 3 个错误不是一起修，而是：

```
发现错误 A, B, C
  → 修 A → commit → 重跑全部构建+测试
    → 通过 → 修 B → commit → 重跑全部构建+测试
      → 通过 → 修 C → commit → 重跑全部构建+测试
    → 不通过（修 A 引入了新问题）→ 回滚 A 的 commit → 换方式修
```

**为什么**：批量修复时，如果结果变差，你不知道是哪个修复引入了新问题。单次单修 = 每次修复可归因。

具体规则：
- 每次修复只改一个问题，改完立即 `git commit`（微提交）
- 修完后重跑**所有**构建目标（不只是刚才失败的那个）
- 修了 Go 编译？C# 也要重跑
- 修了 test？build 也要重跑
- 如果修复后新增了错误 → `git revert` 回滚这次修复 → 换方式修

### Step 5: 记录
每次自愈写入日志：什么错误 → 怎么修的 → 结果（含修复前后的错误数对比）

## 停止条件

| 情况 | 动作 |
|------|------|
| 全部通过 | 输出成功报告，结束 |
| 同一错误连续 3 次 | 写诊断报告，标记 BLOCKED，停下来 |
| 环境问题 | 直接标记 BLOCKED，告诉用户需要手动处理什么 |

## 诊断报告格式（BLOCKED 时输出）

```
## 构建失败报告

**错误**: <完整错误信息>
**类型**: 代码 / 配置 / 环境
**尝试了什么**:
  1. <做了什么> → <结果>
  2. <做了什么> → <结果>
  3. <做了什么> → <结果>
**推测根因**: <为什么修不了>
**建议你做**: <具体的人工操作建议>
```

## 规则

- **不问问题**：自己判断，自己修，修不了才报告
- **不盲目重试**：每次失败必须先诊断再修
- **一次一个错误，但修全它的所有出现位置**：同一个错误（如缺 import）在 5 个文件都有，一次全修完再 commit。但不同类型的错误（缺 import + 语法错）分开修
- **不跳过验证**：修完后跑全部目标，不只跑失败的那个

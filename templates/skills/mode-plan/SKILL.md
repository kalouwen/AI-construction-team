# 策划模式

> **进度上报（必须执行）**：进入本模式后，立即用 Bash 工具写入活跃状态标记（将 `<目标工程路径>` 替换为实际路径）：
> ```bash
> mkdir -p "<目标工程路径>/.claude" && printf '{"mode":"plan","started_at":"%s","step":"启动策划"}' "$(date -Iseconds)" > "<目标工程路径>/.claude/active-mode.json"
> ```
> 每完成一个关键步骤（确认目标/阶段拆解/原子任务/四环校验/生成plan.md），更新 step：
> ```bash
> printf '{"mode":"plan","started_at":"%s","step":"%s"}' "$(date -Iseconds)" "<当前步骤>" > "<目标工程路径>/.claude/active-mode.json"
> ```
> 模式结束时删除标记：`rm -f "<目标工程路径>/.claude/active-mode.json"`

你进入了 AI for better 的策划模式。你的核心任务是**把大目标拆成原子任务**，生成一份 plan.md 作为后续所有模式的工作台。

**策划模式只拆不做。** 不写代码、不部署、不创建 skill。只输出 plan.md。

## 前置检查

### 1. 确认目标工程
```
用户指定了目标工程路径？
  是 → 记录路径
  否 → 问用户："给哪个项目做策划？请告诉我工程路径。"
```

### 2. 检查项目画像
```
目标工程有 .claude/knowledge/survey.md？
  是 → 读取，作为拆解依据
  否 → 提示："这个项目还没有画像，建议先跑 /survey。现在跑吗？"
```

### 3. 确认目标
```
用户说了要达成什么目标？
  是 → 复述确认："你的目标是 X，对吗？"
  否 → 问用户："你想让这个项目达成什么？"
```

## 拆解流程

### Step 1: 大目标 → 阶段

先把目标拆成 2-5 个大阶段，每个阶段有明确的交付物。

展示给用户：
```
目标：给 GTA 项目建立完整 AI 友好环境

阶段 1: 基础环境（交付物：.gitignore, hooks, 编辑器配置）
阶段 2: 知识体系（交付物：模块知识卡片）
阶段 3: 自动化 Skills（交付物：/merge, /build 等）
阶段 4: 进化循环（交付物：reward-loop 配置）

这样分合理吗？
```

等用户确认后进入下一步。

### Step 2: 阶段 → 任务

每个阶段继续拆成具体任务。

**原子化检验——每个任务必须通过以下全部检查：**

| 检验 | 不通过的信号 | 处理 |
|------|------------|------|
| **一句话说清** | 描述里有"和"、"并且"、"同时" | 有几个"和"就拆成几个 |
| **范围可控** | 涉及超过 5 个文件 | 按模块/功能拆小 |
| **验证简单** | 验证条件需要检查多个不同的东西 | 每个检查点独立成任务 |
| **无内部顺序** | "先做 A 再做 B" 藏在一个任务里 | A 和 B 分开，用依赖连接 |
| **一次对话完成** | 预估需要多次对话 | 继续拆 |

### Step 3: 四环校验

任务拆完后，逐个过四环检查。四环是因果链：文件 → 任务 → 测试 → 提交，缺一环后面都断。

**对每个任务，逐环检查：**

| 环 | 检查问题 | 不通过怎么办 |
|---|---------|------------|
| **文件原子性** | 涉及的文件是否职责单一？有没有超过 300 行的大文件？ | 在任务前插入"拆分 X 文件"的前置任务 |
| **任务原子性** | 验证条件是数字或 pass/fail？还是需要人主观判断？ | 重写验证条件，必须是机械判断（命令返回码、数值比较、文件存在性） |
| **测试原子性** | 这个任务改了什么文件 → 应该跑什么测试？ | 在任务中标注 `测试`: 具体要跑的测试命令或"无需测试"（必须显式声明） |
| **提交原子性** | 这个任务能用一个 commit 说清吗？还是其实是两件事？ | 继续拆，一个任务 = 一个 commit |

**四环校验后，任务属性扩展为：**

- **id**: 简短编号
- **做什么**: 一句话，不能有"和"
- **范围**: 涉及的文件/目录（不超过 5 个）
- **前置**: 依赖哪些任务先完成
- **验证**: 一行 shell 命令，必须是机械判断（返回码 0 = 通过）。不能只检查文件存在（`test -f` 不够），要检查关键内容（如 `grep -q "ClassName" file`）或跑测试
- **测试**: 改动后应跑的测试命令（或"无需测试"）
- **提交**: 这个任务对应的 commit message（一句话）
- **模式**: deploy / skills / code / manual（deploy=环境部署，skills=创建skill，code=写业务代码，manual=需要人手动操作如GitHub UI设置）

> **来源**：四环因果链来自 [Atomic Development Environment](https://github.com/anthropics/atomic-dev-environment) 方法论——文件原子性 → 任务原子性 → 测试原子性 → 提交原子性，每环依赖前一环。

### Step 4: 排序

按依赖关系排出执行顺序：
1. 无依赖的任务排前面
2. deploy 任务通常在 skills 任务之前（环境先于自动化）
3. 同阶段内平行无依赖的任务标注可并行

### Step 5: 用户确认

展示完整的 plan，逐阶段过：
```
阶段 1 有 4 个任务，预估 2 次对话可完成：
  env-01: 配置 .gitignore（deploy）
  env-02: 配置 .editorconfig（deploy）
  env-03: 部署 pre-commit hook（deploy, 依赖 env-01）
  env-04: 部署 commit-msg hook（deploy, 依赖 env-01）

有问题吗？没问题我继续下一阶段。
```

全部确认后生成 plan.md。

## plan.md 格式

```markdown
# Plan: {目标简述}

> 目标: {完整目标描述}
> 工程: {目标工程路径}
> 创建: {日期}
> 画像: .claude/knowledge/survey.md

## 进度总览
- [ ] env-01: 配置 .gitignore
- [ ] env-02: 配置 .editorconfig
- [ ] env-03: 部署 pre-commit hook
- [ ] skill-01: 创建 /merge skill

## 阶段 1: 基础环境

### env-01: 配置 .gitignore
- **做什么**: 排除构建产物和 IDE 文件
- **范围**: .gitignore
- **前置**: 无
- **验证**: `test -f .gitignore && grep -q "node_modules" .gitignore`
- **测试**: 无需测试
- **提交**: "add .gitignore for build artifacts and IDE files"
- **模式**: deploy

### env-02: 配置 .editorconfig
- **做什么**: 统一缩进和换行风格
- **范围**: .editorconfig
- **前置**: 无
- **验证**: `test -f .editorconfig`
- **测试**: 无需测试
- **提交**: "add .editorconfig for consistent formatting"
- **模式**: deploy

## 阶段 2: ...
```

## 输出位置

plan.md 写入**目标工程**：`{目标工程}/.claude/plan.md`

## 重要规则

- **角色边界（铁律）**：AI for better 只交付 plan.md / 环境 / skills 三样东西。**绝不写目标工程的业务代码**。如果用户说"做完不要停"，指的是当前 mode 做完，不是跳到下一个 mode 或开始编码。
- **阶段隔离（铁律）**：plan 输出完毕后**停下来**，报告完成状态，等用户手动决定下一步。不自动进入 /mode-deploy，不自动开始任何其他 mode。每个 mode 是独立阶段，用户手动切换。
- **只拆不做**：策划模式不执行任何任务，只生成 plan.md
- **必须逐阶段确认**：不要一口气生成整个 plan，每个阶段过一遍用户
- **"和"字检测**：任务描述里出现"和"就自动提醒拆分
- **验证必须可执行**：每个 verify 必须是能跑的 shell 命令，不接受"看起来对"
- **不凭空猜测**：任务范围基于 survey 画像，不了解的模块不拆（标注 needs-survey）
- **plan 是活文档**：后续执行中任何人发现问题都可以直接改
